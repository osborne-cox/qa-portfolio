# QA Portfolio

Портфолио тестовой документации на учебном стенде [demoqa.com](https://demoqa.com). Цель — показать не «я знаю слово регрессия», а как я думаю: где искать границы, как отличить фичу от бага, как описать находку так, чтобы разработчик понял её с первого раза, а не переспрашивал три раза.

Всё писалось на основе фактического поведения demoqa.com. Там, где сайт ведёт себя не так, как ждёшь, это отдельно отмечено в колонке «Фактический результат» — именно расхождения с «ожидаемым» и есть пища для багов.

## Как читать тест-кейсы

У каждого кейса свой ID (`TC-001-04`), приоритет, тип (позитивный / негативный / граничный), предусловие, тестовые данные, шаги и два результата: **Ожидаемый** (как должно быть) и **Фактический** (как есть на demoqa). Статус Pass/Fail проставлен по факту прогона. Где demoqa делает «технически работает, но по-человечески странно» — это помечено как наблюдение, а не как ложный Fail.

Приоритеты:
- **High** — основной сценарий или частая ошибка пользователя;
- **Medium** — типовой негативный кейс;
- **Low** — редкий граничный ввод.

## Сводка находок

Если в двух словах: функционально demoqa работает (CRUD, чекбоксы, каскады — всё на месте), а сыпется на валидации и вёрстке. Поля принимают цифры вместо имени, 500+ символов и пробелы без trim; числовые поля таблицы глотают буквы; а на большой форме реклама буквально перекрывает кнопку отправки. Для боевого продукта это были бы реальные дефекты — здесь они удобный материал, чтобы показать, что искать.

## Оговорка про честность

demoqa — учебный стенд, его специально не всегда чинят, поэтому описанные баги (особенно BUG-001 с рекламой) воспроизводятся стабильно и у многих. К баг-репортам по-хорошему прикладываются скриншоты — в этом наборе для них оставлены пометки; при живом прогоне в браузере их стоит добавить.
