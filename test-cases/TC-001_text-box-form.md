# TC-001 — Форма «Text Box»

**Страница:** https://demoqa.com/text-box
**Объект тестирования:** форма с полями Full Name, Email, Current Address, Permanent Address и кнопкой Submit.
**Окружение:** Firefox / Windows 11, разрешение 1920×1080.

Краткая вводная. Формально валидируется только поле Email (проверка формата). Остальные поля не обязательны и не валидируются — это видно по результатам кейсов ниже. В колонке «Фактический результат» зафиксировано поведение сайта, в «Ожидаемом» — как форма вела бы себя по-хорошему. Расхождения между ними и есть самое интересное.

Обозначения приоритета: High — основной бизнес-сценарий, Medium — типовая ошибка пользователя, Low — редкий/граничный ввод.

---

## TC-001-01 — Отправка корректно заполненной формы (happy path)

| | |
|---|---|
| **Приоритет** | High |
| **Тип** | Позитивный |
| **Предусловие** | Открыта страница /text-box, все поля пустые |

**Тестовые данные:**
- Full Name: `Danil Ivanov`
- Email: `danil.ivanov@example.com`
- Current Address: `Tverskaya St, 7, Moscow`
- Permanent Address: `Lenina St, 14-22, Kazan`

**Шаги:**
1. Заполнить Full Name значением из тестовых данных.
2. Заполнить Email.
3. Заполнить Current Address.
4. Заполнить Permanent Address.
5. Нажать Submit.

**Ожидаемый результат:** под формой появляется блок Output с рамкой, в нём выводятся ровно те значения, что были введены: Name, Email, Current Address, Permanent Address.

**Фактический результат:** соответствует ожидаемому. Блок Output отображается, все 4 значения совпадают с введёнными.

**Статус:** Pass

## TC-001-02 — Email без символа `@`

| | |
|---|---|
| **Приоритет** | High |
| **Тип** | Негативный |
| **Предусловие** | Открыта страница /text-box |

**Тестовые данные:** Email = `danilexample.com`, остальные поля заполнены валидно.

**Шаги:**
1. Заполнить Full Name и оба адреса валидными значениями.
2. В поле Email ввести `danilexample.com` (без `@`).
3. Нажать Submit.

**Ожидаемый результат:** форма не отправляется, поле Email подсвечивается как невалидное, рядом сообщение о неверном формате почты.

**Фактический результат:** поле Email получает красную рамку (класс `field-error`), блок Output не формируется. Текстового сообщения об ошибке нет — только цвет рамки. Пользователь без внимательного взгляда может не понять, что именно не так.

**Статус:** Pass (валидация срабатывает, но обратная связь скудная — см. примечание)

> Примечание: отсутствие текста ошибки — кандидат в минорный UX-баг, но в рамках этого кейса проверяем сам факт блокировки.

## TC-001-03 — Отправка полностью пустой формы

| | |
|---|---|
| **Приоритет** | Medium |
| **Тип** | Негативный |
| **Предусловие** | Открыта страница /text-box, ни одно поле не заполнено |

**Шаги:**
1. Не заполняя ничего, нажать Submit.

**Ожидаемый результат:** если поля обязательны — подсветка обязательных полей и сообщения «поле обязательно». Если необязательны — это должно быть осознанным решением, а не побочным эффектом.

**Фактический результат:** никаких ошибок не возникает, обязательность полей не проверяется вообще. Блок Output либо не появляется, либо появляется пустым. То есть «отправить ничего» — валидное действие.

**Статус:** Pass по факту поведения / Fail по здравому смыслу формы регистрации (расхождение зафиксировано как наблюдение).

## TC-001-04 — Имя из одних цифр

| | |
|---|---|
| **Приоритет** | Medium |
| **Тип** | Негативный / граничный |
| **Предусловие** | Открыта страница /text-box |

**Тестовые данные:** Full Name = `1234567890`.

**Шаги:**
1. В Full Name ввести `1234567890`.
2. Email — валидный, адреса заполнить.
3. Нажать Submit.

**Ожидаемый результат:** имя из одних цифр отклоняется либо помечается как подозрительное (имя не может состоять только из цифр).

**Фактический результат:** значение принимается без вопросов и выводится в Output как есть. Валидации содержимого имени нет.

**Статус:** Fail (по дизайну валидации) — баг класса «отсутствует проверка формата имени». Тесно связан с BUG-002.

## TC-001-05 — Очень длинный текст в полях (500+ символов)

| | |
|---|---|
| **Приоритет** | Low |
| **Тип** | Граничный |
| **Предусловие** | Открыта страница /text-box |

**Тестовые данные:** Full Name и Current Address — строки по 550 символов (повтор `A` и осмысленный длинный адрес).

**Шаги:**
1. Вставить в Full Name строку из 550 символов.
2. Вставить в Current Address строку из 550 символов.
3. Email — валидный.
4. Нажать Submit.

**Ожидаемый результат:** у полей есть разумный лимит длины (например, 50–100 для имени), ввод сверх лимита обрезается или отклоняется.

**Фактический результат:** ограничения по длине нет (атрибут `maxlength` отсутствует). Все 550 символов принимаются и выводятся в Output, длинная строка растягивает/ломает вёрстку блока вывода.

**Статус:** Fail — баг «нет ограничения длины полей». Зафиксировано в BUG-002.

## TC-001-06 — Только пробелы в полях

| | |
|---|---|
| **Приоритет** | Low |
| **Тип** | Негативный / граничный |
| **Предусловие** | Открыта страница /text-box |

**Тестовые данные:** Full Name = `"     "` (5 пробелов), Current Address = `"   "` (3 пробела).

**Шаги:**
1. Ввести в Full Name только пробелы.
2. Ввести в Current Address только пробелы.
3. Email — валидный.
4. Нажать Submit.

**Ожидаемый результат:** строки из одних пробелов трактуются как пустые (trim), поведение совпадает с пустым полем.

**Фактический результат:** пробелы не обрезаются (нет trim). Поле «из пробелов» считается заполненным, значение уходит в Output как пустая на вид строка. Визуально в Output появляется строка `Name :` без видимого значения.

**Статус:** Fail — отсутствует trim/нормализация ввода.

## TC-001-08 — Валидное имя на кириллице + email с IDN-доменом

| | |
|---|---|
| **Приоритет** | Low |
| **Тип** | Позитивный / локализация |
| **Предусловие** | Открыта страница /text-box |

**Тестовые данные:** Full Name = `Данил Иванов`, Email = `данил@почта.рф`.

**Шаги:**
1. Ввести кириллическое имя.
2. Ввести email с кириллическим доменом (IDN).
3. Нажать Submit.

**Ожидаемый результат:** кириллическое имя принимается; по email с IDN поведение нужно зафиксировать (поддержка IDN не гарантируется).

**Фактический результат:** имя на кириллице принимается и выводится корректно. Email `данил@почта.рф` помечается красной рамкой — встроенная регулярка не считает кириллический IDN валидным. Пограничный случай, отмечаю как ожидаемое ограничение, а не баг.

**Статус:** Pass (с оговоркой по IDN)
